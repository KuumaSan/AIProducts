每天早上 7:30（Asia/Shanghai），为 Telegram 频道 AI Product News 生成一份 AI 日报，并主动发送到 Telegram 频道 -1003810364096。

任务目标分三部分：
A. 生成并发送日报
B. 更新工作区里的项目追踪表 tracking/ai-projects.csv
C. 如果文件有变化，则自动提交并推送到 GitHub 仓库

A. 日报要求：
1. 先使用 web_search（Brave）查找最近 72 小时内新出现或热度明显上升的 AI 产品；如果当天结果不足，再放宽到最近 7 天。
2. 除了媒体新闻，也要主动关注 GitHub 上的 AI 项目热度变化，尤其是 star 增长明显、短期被大量讨论或快速进入开发者视野的仓库。
3. 如果某个 GitHub AI 项目没有大新闻，但 star 激增、issue/讨论活跃或生态扩张明显，也可以纳入日报。
4. 优先选 3-5 个产品，避免泛泛而谈的行业新闻。
5. 先读取 tracking/ai-projects.csv，避免重复发送已经讲过但没有实质新进展的产品。
6. 如果某个产品已经在 tracking/ai-projects.csv 中出现过，但最近出现了明确的新功能、新融资、新用户增长、新合作、重大争议、收购、转型、停更等实质变化，可以作为"更新项"发送。
7. 如果某个产品只是再次出现在新闻里、但没有明显新增信息，就不要再次发日报正文；最多只在内部更新 CSV。
8. 对每个产品必须回答：
   - 它解决的需求是什么
   - 它为什么会火
   - 这个需求是真需求、半真需求还是伪需求
   - 前景如何
9. 必须保持批判性思维，不能一味吹捧；如果是噱头大于价值，要明确说出来。
10. 输出格式必须适合 Telegram 手机版阅读：
   - 第 1 条消息：总览（日期、今日一句话判断、今天选了哪些产品）
   - 后续每个产品单独 1 条消息
   - 每条产品消息都控制在适合手机阅读的长度
   - 每条产品消息最后一行单独放“来源：”和 URL，方便 Telegram link preview 正常显示
11. 不要把多个产品塞进一条消息里。
12. 对于已跟踪产品的更新项，在标题里明确标注“更新”，例如：Cursor Automations（更新）。
13. 如果当天没有足够值得写的新产品，也没有足够重要的旧产品更新，就发一条简短总览，明确说明“今天没有特别值得跟踪的新 AI 产品发布或重要更新”，不要硬凑。

B. 追踪表更新要求：
1. 表格文件是 tracking/ai-projects.csv。
2. 如果日报中出现了新的项目，就追加一行。
3. 如果项目已经存在，就更新这些列：last_updated、why_hot、verdict、status、last_checked、source_url、notes。
4. domain 必须从这些类别中选最合适的一个：
   - AI 编程 / 开发工具
   - 多模态生成 / 创意工具
   - AI Agent / 自动化
   - 搜索 / 研究工具
   - 医疗 / 健康 AI
   - 企业服务 / SaaS
   - 销售 / 营销
   - 教育
   - 基础模型 / 平台
   - 其他
5. CSV 列顺序必须保持不变：
   project,domain,first_seen,last_updated,need,why_hot,verdict,status,last_checked,source_url,notes
6. status 默认写“活跃”，除非你有明确证据表明项目停更、关闭、被收购或转型。
7. first_seen 只在新项目首次加入时填写当天日期，旧项目不要改它。
8. 更新 CSV 时，保持合法 CSV 格式，必要时正确使用引号。
9. 如果项目主要是从 GitHub 热度信号发现的，在 notes 中简要注明这是“GitHub 热度发现”，并简述为什么值得跟踪。

C. Git 自动提交要求：
1. 在完成 CSV 更新后，检查 git 工作区是否有 tracking/ 或 reports/ 下的变化。
2. 如果有变化，执行：
   - git add tracking reports
   - git commit -m "Daily AI tracker update YYYY-MM-DD"
   - git push origin master
3. 如果没有变化，就不要强行提交。
4. 不要提交任何密钥、环境变量或无关临时文件。

发送方式：
- 使用 message 工具主动发送到 channel=telegram, target=-1003810364096
- 先发 1 条总览，再逐条发产品消息
- 完成日报发送、CSV 更新和 git push（如有变化）后，最终文本回复只输出 NO_REPLY，避免重复投递
